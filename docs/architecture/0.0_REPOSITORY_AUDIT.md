# Sprint 0.0: Repository Audit

## 1. Executive Summary
This document provides a high-level inventory of the FoodieGo monorepo, encompassing backend services, shared packages, frontend applications, and infrastructure configurations. This audit satisfies the Rule 1 constraint of the Principal Architect guidelines: "Read Everything, Not Only Business Logic."

## 2. Backend Services (`apps/`)

| Service Name | Description | DB Technology | Identified Issues |
| :--- | :--- | :--- | :--- |
| `identity-service` | Auth, user profiles, RBAC, admin management. | Raw `pg` SQL | Directly writes to `users` using inline SQL. No ORM. |
| `restaurant-service`| Restaurant profiles, menus, categories. | Raw `pg` SQL | Contains routes for reading menus, but missing POST routes for Merchant creation. |
| `order-service` | Cart, Checkout, Delivery, Order lifecycle. | Raw `pg` SQL | **Critical DDD Violation:** `checkout.service.js` directly queries `inventory_stock` using `FOR UPDATE`. |
| `payment-service` | Mock payment integration, webhooks. | Raw `pg` SQL | Very thin layer. Idempotency exists but saga compensation is incomplete. |
| `inventory-service` | Stock management. | Raw `pg` SQL | Exists but bypassed entirely by `order-service` doing direct DB access. |
| `gateway` | API Gateway (Express). | None | Standard proxy routing. |
| `food-service` | Unknown/Legacy. | Unknown | High likelihood of being dead code/duplicate of `restaurant-service`. |

## 3. Shared Packages (`packages/`)

The repository suffers from severe package sprawl and duplication.

### 3.1 SDKs & Types
*   `api-sdk`: Internal API caller.
*   `platform-sdk`: Common wrappers.
*   `contracts`: RabbitMQ schemas and events.
*   `types`, `shared-types`: Duplicate type definition packages.

### 3.2 Observability Sprawl
There are four distinct observability packages:
*   `logger`
*   `logging`
*   `metrics`
*   `otel`
**Action Required:** Merge into a single `@foodiego/observability`.

### 3.3 Infrastructure Utilities
*   `rabbit`: AMQP connection handling.
*   `retry`: Outbox/Inbox patterns.
*   `database`: Exists but mostly empty (services are importing `pg` directly).
*   `persistence`: Unknown utility.
*   `problem`: RFC 7807 error format.
*   `core`, `utils`, `shared-utils`: Utility sprawl. Merge required.

## 4. Frontend Application (`apps/web`)
*   **Tech Stack:** React, Vite, TailwindCSS, Zustand, React Query.
*   **Structure:** Has `apps/web/src/pages`, `apps/web/src/store`.
*   **Status:** Missing distinct portals for Merchant and Admin. The customer flow lacks E2E integration with the backend saga.

## 5. Infrastructure & Tooling
*   **Database:** A single shared PostgreSQL database (`foodiego`), which leads to the cross-domain queries seen in `order-service`.
*   **Message Broker:** RabbitMQ.
*   **CI/CD:** Basic GitHub Actions exist.
*   **Docker:** Docker Compose used for local orchestration.

## 6. Initial Architecture Fitness Assessment (High-Level)
*   **DDD Violations:** High. Services are querying other domains' tables directly (e.g., Order -> Inventory).
*   **Duplication:** High. Too many utility and logger packages.
*   **Dead Code:** `food-service` appears deprecated in favor of `restaurant-service`.
