# FoodieGo v0.6.0-beta Release Notes

## 🎯 Release Objective
This release (`v0.6.0-beta`) transitions the system from "works on my machine" to **"Production-Ready and Evidence-Backed."** It validates the robustness of the core event-driven architecture, distributed tracing, idempotency, metrics, and data integrity under load. 

We focused on resolving real-world microservice complexities such as:
- Chaos engineering (RabbitMQ crashes and Event Replays).
- Observability (Distributed tracing and metrics via OpenTelemetry/Prometheus/Grafana).
- Data Consistency (Saga pattern, Inbox/Outbox).

---

## ✅ Accomplishments & Evidence

### 1. Robust Distributed Tracing (Tempo & OpenTelemetry)
**Issue:** Previously, `X-Trace-Id` headers were lost between the API Gateway, Order Service, and RabbitMQ events.
**Resolution:** Properly injected OpenTelemetry propagation into the Express middleware and RabbitMQ AMQP envelopes. 
**Evidence:**
We created a verification script (`scripts/verify-trace.js`) that performs a full checkout flow, queries the Grafana Tempo API directly, and parses the distributed spans:
- `order-service` span (Checkout initialization).
- `inventory-service` span (RabbitMQ event consumed and reservation handled).
Both services are visible in a unified trace across HTTP and AMQP boundaries.

### 2. Comprehensive Prometheus Metrics
**Issue:** `inventory-service` metrics were missing, and metrics weren't automatically validated.
**Resolution:** Updated `prometheus.yml` to scrape the newly containerized inventory service. Created `scripts/verify-metrics.js`.
**Evidence:**
Our automated verification queries `http://localhost:9090/api/v1/query?query=up` and ensures all 5 critical microservices are actively scraped and returning `1`.

### 3. Saga Pattern & Event-Driven Reliability (Inbox/Outbox)
**Issue:** RabbitMQ dropped messages due to un-matched topics between producer (`foodiego.events`) and consumer (`foodiego.orders.events`).
**Resolution:** Standardized the RabbitMQ Topology to a direct exchange configuration (`EventRegistry`).
**Evidence:** 
- The Outbox table now processes payloads via the Node.js Dispatcher gracefully.
- The Inbox pattern now guarantees idempotency of duplicate incoming messages. 

### 4. Chaos Testing: RabbitMQ Crash Recovery
**Issue:** If RabbitMQ crashed, in-flight messages were permanently lost.
**Resolution:** Leveraging the Inbox/Outbox pattern, our Dispatcher worker actively scans `PENDING` outbox events.
**Evidence:** We brought the RabbitMQ container down (`docker stop foodiego-rabbitmq`), simulated checkout requests (which successfully wrote to `outbox_events`), and restarted RabbitMQ. The Dispatcher successfully pushed the backlogged events and the Saga completed!

### 5. Inventory Reservation Under High Concurrency
**Issue:** Data mismatch between successful reservations and concurrent reserved stock capacity.
**Resolution:** Validated that optimistic locking accurately prevents double-booking while TTL processes gracefully release expired holds.
**Evidence:** `scripts/verify-inventory.js` successfully ran alongside `k6-checkout.js` load testing, verifying that `total_quantity >= reserved_quantity` remains universally true under peak load.

## 🛠 Reproducible CI/CD Pipeline
We established a strict sequence for testing any local environment or CI runner:

```bash
# 1. Reset Database State
node scripts/reset-db.js

# 2. Seed All Data (Users, Restaurants, Inventory, Load Test Data)
node scripts/seed/all.js

# 3. Verify Distributed Traces
node scripts/verify-trace.js

# 4. Verify Prometheus Metrics
node scripts/verify-metrics.js
```

These scripts execute sequentially without failure, proving that our infrastructure configuration is robust and repeatable.
